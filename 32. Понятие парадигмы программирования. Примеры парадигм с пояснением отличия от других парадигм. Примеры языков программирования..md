***Парадигма*** - совокупность идей, определяющих подход к реализации программ. Парадигма определяет стиль написания кода, а не организацию процесса разработки. Своего рода философия написания кода. (Способ рассуждать о решении задачи, о будущей программе)

---
### Примеры парадигм с пояснением отличия от других парадигм

Формально языки делятся на имеративные и декларативные. Если быть точнее то правильнее говорить о степени императивности и декларативности (так как почти все языки на данный момент мультипарадигменные)

***Императивное программирование*** - подход, при котором программа представлена как последовательность инструкций, которые вычислитель выполняет полностью соответствуя тому, что написано.
Изменение значений в памяти ЭВМ
Примеры: Cobol, Basic

***Декларативное программирование*** - подход, при котором программа представляется как последовательность запросов и команд вычислителю, которое он обрабатывает так, как считает нужным. Программа описывает логику решения задачи а не шаги, которые необходимо выполнить.
Декларативные программы не используют ни переменные, ни операторы присваивания, дабы избежать появления потенциальных ошибок (гонка за ресурсы)

С точки зрения разработчика ПО:
- Самые имеративные языки - язык Ассемблера
- Самые декларативные - конфигурации систем сборки (`CMakeLists.txt`) и языки запросов к БД

#### **Структурное программирование**
Стиль структурного программирования расширяет базовый императивный принцип за счет специальных управляющих структур: последовательность (шаги идут друг за другом), ветвления, циклы

Этот подход основан на желании ограничить или полностью избежать операторов перехода (goto), которые делают императивно разработанный код излишне сложным

Решаемые задачи:
- сокращение ошибок
- упрощение отладки, модификации, сопровождения
C, Python

#### **Процедурное программирование**
Процедурный подход делит программу на более мелкие подзадачи, которые по отдельности описываются в коде

Описание подпрограмм (процедур, функций)
- Декомпозиция 
- Переиспользование кода

Программа как последовательность изменений состояния вычислителя
Assembler, Fortran

#### **Модульное программирование**
Это надстройка над процедурным. Группировка процедур и данных в модули (файлы, пакеты). Главная цель - сокрытие информации. Мы прячем внутренние детали реализации (глобальные переменные, вспомогательные функции) внутри модуля, а наружу выставляем только то, что нужно

Модульное программирование вводит понятие области видимости на уровне файла/модуля
Algol-68, Pascal

#### **Логическое программирование**
Программа представляет собой набор фактов и правил, система сама строит решение с использованием правил логики. Решение задачи - доказательство теоремы. 
Нет привычного понятия "функция". Есть предикаты (утверждения, которые истины или ложны)

- Системы ИИ используют свои факты и правила для каждого запроса. Это одно из основных приложений логического программирования
- Хорошо подходит для анализа содержимого БД
- Логические программы могут сортировать большие объемы данных, анализировать результаты и делать прогнозы
Prolog, Datalog

#### **Функциональное программирование**
Парадигма программирования, в которой процесс вычисления трактуется как вычисление значений функций в математическом понимании последних (чистые функции)

**Чистые функции**
- Функция зависит только от своих аргументов, не ссылаясь на глобальные состояния. То есть всегда возвращает один и тот же результат для одних и тех же входных данных.
- Не вызывает побочных эффектов. Не меняет глобальные переменные, не пишет в файлы, не пишет в консоль. Она только возвращает новое значение.
Функционально программирование предназначено для отделения логического описания от его фактической интерпретации. В общем главная цель - запретить изменение состояния, чтобы не допустить ошибок.
LISP, Haskell, Ocaml

#### ООП (Объектно Ориентированное Программирование)
Подход, при котором программа рассматривается как набор объектов, взаимодействующих друг с другом. У каждого есть свои свойства и поведение
ООП появилось из-за увеличения объема кода

**Принципы SOLID**
- ***Single Responsibility*** (принцип единственной ответственности)
	Каждый объект должен отвечать только за одну операцию
- ***Open-Closed***
	Объекты должны быть открыты для расширения, но закрыты для модификации
- ***Liskov Substitution***
	Любые наследованные объекты должны уметь заменять своего родителя без последствий
- ***Interface Segregation*** (принцип разделения интерфейсов
	Объекты не должны зависеть от методов, которые они не используют
- ***Dependency Inversion*** (Принцип инверсии зависимостей)
	Родитель не должен зависеть от детей

Задачи:
- Лучшая расширяемость
- Эффективная поддержка программ
Java, JS, C#
