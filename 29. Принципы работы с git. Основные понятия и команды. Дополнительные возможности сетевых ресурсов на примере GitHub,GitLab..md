
***Git*** - распределенная система контроля версий, позволяющая сохранять изменения, внесенные в файл, которые хранятся в репозитории. Сами изменения сохраняются в виде коммитов. Значительно облегчается взаимодействие с другими разработчиками: несколько человек могут работать над одним репозиторием одновременно, сохраняя свои изменения.

### Основные понятия

***Рабочая директория*** - директория на локальном компьютере, в котором находится проект. По факту место для работы (создавать, удалять, изменять файлы в процессе разработки)

***Index file*** - двоичный файл, содержащий список всех файлов, которые будут включены в следующий коммит. Пользователь сам может выбирать, какие файлы включать в коммит, а какие нет

***Локальный репозиторий*** - это база данных Git на локальном компьютере, которая содержит все версии и историю изменений файлов проекта. Позволяет работать с проектом без подключения к удаленному репозиторию

***Удаленный репозиторий*** - база данных Git, расположенная на удаленном сервере, к которому можно обращаться через сеть. Позволяет синхронизировать изменения между локальными копиями репозитория разных разработчиков

***Blob (binary large object)*** - содержимое файла (копия всех данных файла для каждой его уникальной версии)

***Commit*** - снимок всего содержимого репозитория на момент сохранения. Для каждого файла хранится ссылка на blob. Если файл меняется и коммитится, то создается новый blob. Если файл не меняется, то коммит сохраняет ссылку на наиболее новый blob для этого файла.

***Ветки*** - отдельные истории коммитов. Позволяет разработчику работать над разными версиями проекта одновременно. Новые ветви получаются путем отведения от уже имеющихся. Истории разных веток развиваются независимо. Изменения одной ветки можно слить в другую.

***Конфликт*** - ситуация, которая происходит при слиянии ветвей, когда один и тот же файл меняется в различных ветках
Чтобы решить конфликт нужно:
- привести файл в котором он возник к желаемому виду
- можно доверится железке и она сама придумает как его решить, но лучше смотреть и менять вручную, так надежнее

**Состояния файлов**:
- **неотслеживаемый (untracked)** - Git о нем не знает
- **изменен (modified)** - файл изменен, но его нет в index file
- **подготовлен (staged)** - файл изменен и присутствует в index file
- **без изменений (unmodified)** - в последнем коммите находится самая актуальная версия

### Основные команды
1) `git init` - создать пустой репозиторий
2) `git clone` - скопировать удаленный репозиторий себе на локальную машину
3) `git add` - добавить изменение в stage
4) `git commit` - сделать коммит изменений, добавленных в stage
5) `git switch` - переключиться на ветвь
	`git switch -c` - создать ветвь и переключиться на нее
6) `git revert <commit>` - откатить изменения через создание нового коммита
7) `git restore` - откатывает unstaged изменения

8) `git push` - отправка изменений в удаленный репозиторий
9) `git fetch` - получение изменений с удаленного репозитория (но не изменяет локальный репозиторий)
10) `git pull` - получение изменений с удаленного репозитория (выполняет `git merge` с вашим локальным репозиторием)
11) `git branch` - для работы с ветками
12) `git checkout` - переключение между созданными ветками и коммитами. HEAD перемещается на указатель другой ветки или коммита
13) `git merge` - объединение изменений из одной ветки в другую
14) `git status` - проверка состояния файлов в рабочей директории
15) `git log` - просмотр изменений в репозитории
16) `git diff` - просмотр изменений между двумя коммитами, ветками или файлами
17) `git reset` - отмена изменений, вернувшись к конкретному коммиту. В отличие от `git revert`, `reset` удаляет комиты из истории, сдвигая указатель ветки назад. `revert` создает новый коммит, отменяющий изменения старого
18) `git rebase` - переносит или объединяет коммиты из одной ветки в другую, переписывая историю для создания линейной (чистой) истории

### Дополнительные возможности сетевых ресурсов на примере GitHub/GitLab

1) **Fork**
	Создание своей версии репозитория на основе уже существующего репозитория. Позволяет контрибьютить (добровольно участвовать в разработке, улучшать или поддерживать ПО чаще всего в open-source проекты) в проекты, не получая прямого доступа к внесению изменений в репозиторий путем pull request'ов

2) **Pull request / merge requests**
	Координация слияния изменений в проектах над которыми работает несколько человек. При forke авторы проекта смотрят на pull request'ы и решают одобрять их или нет.

 3) **CI/CD**
	 **CI (Continues Integration - непрерывная интеграция)** - практика постоянного слияния изменений в основную ветвь и проверки того, что все работает как ожидается
	 Включает в себя:
	 - автоматическая сборка проекта и запуск тестов
	 - работа линтеров (линтер - инструмент автоматического статического анализа кода на наличие синтаксических ошибок)
	 Результат: гарантия того, что в основной ветке всегда находится рабочий код

	**CD (Continued Deployment - продолжение развертывания)** - практика постоянного автоматического развертывания (публикования) артефактов проекта
	- Успешная сборка (бинарный файл, докер образ) загружается в репозиторий
	- Если все этапы CI прошли успешно, код автоматически развертывается на Production без участия человека
