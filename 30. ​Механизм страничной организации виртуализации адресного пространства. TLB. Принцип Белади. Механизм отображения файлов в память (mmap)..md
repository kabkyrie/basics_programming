
### Механизм страничной организации виртуализации адресного пространства процесса

***Страничная организация*** - метод управления памятью, при котором адресное пространство процесса (виртуальная память) делится на блоки фиксированного размера, называемые страницами

- ***Страница*** - минимальная область памяти, которая может быть запрошена процессором у ОС (минимальная единица выделения памяти). Обычно 4096 байт
- **Физическая память** тоже делится на блоки такого же размера, которые называются ***физическими кадрами (frames)***

С точки зрения процесса, память выглядит как непрерывный массив адресов. Однако ОС создает абстракцию виртуальной памяти. Для каждой виртуальной страницы возможны несколько состояний:
- **Mapped** - странице сопоставлен реальный физический блок (frame) в RAM
- **Swapped** - данные страницы были вытеснены из RAM на диск из-за нехватки оперативки
- **Lazy mapping** - страница как бы существует в адресном пространстве процесса, но физическая память под неё ещё не выделялась, так как процесс к ней еще ни разу не обращался

**Page Fault** - прерывание, которое генерирует MMU, если процесс обращается к странице, которая не была отмаплена в физическую память RAM. Обработкой page fault занимается ядро ОС - оно размещает страницу в RAM, при необходимости выгружая другие страницы на диск (вытеснение)

Каждый процесс имеет свою таблицу адресов страниц

### Трансляция адресов. TLB

***Трансляция адресов*** - отображение виртуальных адресов в физические в рамках механизма виртуальной памяти.

Соответствие физических адресов виртуальным адресам хранится в **таблице страниц**

***Таблица страниц*** - структура данных, используемая системой виртуальной памяти в ОС для хранения соответствия виртуальных адресов физическим адресам. Каждый процесс имеет свою собственную таблицу страниц

***TLB (Translation Lookaside Buffer)*** - специальный быстрый кэш внутри MMU, хранящий недавние трансляции адресов

Трансляцией адресов занимается MMU - компонент CPU
Когда процесс обращается к странице:
1) MMU получает виртуальный адрес, из него получает номер страницы и **смещение**
2) Ищет страницу в TLB
3) Если не находит, смотрит в таблице страниц (page tables)
4) Если страница отображена в физическую память, то может сразу получить искомые данные
5) Если процесс еще не обращался к этой странице, сгенерируется page fault
6) Если страница выгружена в swap, сгенерируется page fault

***Смещение*** - адрес внутри самой страницы, указывает на то, что должно быть выполнено (то есть в странице содержится какое-то количество инструкций и смещение страницы показывает, какая конкретно инструкция нам нужна)

### Стратегии вытеснения. Принцип Белади и аномалия Белади

Стратегия вытеснения определяет порядок своппинга страниц (как выбрать страницу для свопа)

**Принцип вытеснения Белади** - вытеснять ту запись, которая дольше всего не понадобится. НО, к сожалению, это алгоритмически не решаемая задача. Используется как эталон для сравнения эффективности других алгоритмов.

Популярные стратегии:
- **FIFO (First-in, First-out)** - вытесняется страница, загруженная раньше всех
- **LRU (Least Recently Used)** - вытесняется страница, к которой дольше всего не обращались

**Аномалия Белади**
Парадоксальный эффект, который может возникать при использовании алгоритма FIFO. Увеличение размера кэша (количества доступных физических фреймов) приводит к увеличению количества page fault при обращении к странице, а не к уменьшению.
Страницы, к которым часто обращаются могут быть выгружены просто потому что они "старые", именно в тот момент, когда они снова нужны.

### Механизм отображения файлов в память (mmap)

***mmap (Memory Mapped Files)*** - механизм, позволяющий проецировать содержимое файла напрямую в виртуальное адресное пространство процесса

**Как это работает:**
1) Процесс открывает файл
	- делает системный вызов `open`
	- получает от ОС файловый **дескриптор**
2) Процесс просит ОС отмаппить файл в его адресное пространство
	- Делает системный вызов `mmap` передавая ранее полученный файловый дескриптор как один из аргументов
	- получает адрес (`char *`)
3) Теперь (с точки зрения процесса) работа с файлом - обычное обращение к указателю в его адресном пространстве
4) С помощью `munmap` процесс может попросить ОС убрать соответствие адресов (разрыв связи адресов и файла)

**Задачи и преимущества:**
- **Оптимизация ввода/вывода**. Один вызов `mmap` заменяет множество вызовов `read/write`
- **Работа с большими файлами**. Можно работать с файлом, который больше, чем доступная RAM. Части файла подгружаются лениво (по мере обращения) и выгружаются обратно автоматически.
- **IPC (Inter-Process Communication - межпроцессорное взаимодействие)**. Если два процесса одновременно сделают `mmap` одного файла, они будут видеть изменения друг друга мгновенно. Это самый быстрый способ обмена данными между процессами.

***Дескриптор*** - абстрактный идентификатор, который ОС выдает процессу, когда тот запрашивает доступ к файлу или другому ресурсу ввода/вывода. С технической точки зрения, дескриптор - это неотрицательное целое число.
Когда вы делаете `mmap`, файловый дескриптор дает понять ядру, какой именно открытый ресурс нужно связать с виртуальной памятью и какие права доступа к нему были получены при открытии

При запуске процесса ОС автоматически открывает для него три стандартных дескриптора:
- `0 (stdin)` - стандартный поток ввода, обычно клавиатура
- `1 (stdout)` - стандартный поток вывода, обычно терминал/экран
- `2 (stderr)` - стандартный поток ошибок
Когда вы открываете собственный файл в программе, `open` обычно возвращает дескриптор `3`
